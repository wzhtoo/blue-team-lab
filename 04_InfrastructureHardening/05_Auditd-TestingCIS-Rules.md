# Testing & Validation

### **Testing the CIS Rules: Step-by-Step**

#### **၁။ `sudo useradd testuser` (The Action)**

ဒါက စက်ထဲမှာ User အသစ်တစ်ခုကို ဖန်တီးလိုက်တာပါ။

* **ဘာဖြစ်သွားမလဲ:** User အသစ်တစ်ခု ဆောက်လိုက်တဲ့အခါ Linux က `/etc/passwd` နဲ့ `/etc/group` ဖိုင်တွေကို သွားပြီး Update လုပ်ပါတယ်။
* **CIS Connection:** ရှေ့မှာ ထည့်ခဲ့တဲ့ Audit Rule ထဲမှာ `-w /etc/passwd -p wa -k identity` ဆိုပြီး ပါခဲ့ပါတယ်။ ဒါကြောင့် ဒီဖိုင်ကို တစ်ယောက်ယောက်က လာပြင်တာနဲ့ Auditd က မှတ်တမ်းတင် (Log) ကျန်ရစ်မှာ ဖြစ်ပါတယ်။

#### **၂။ `sudo ausearch -k identity | tail -n 10` (The Verification)**

ဒါကတော့ Audit logs တွေထဲမှာ "Key" နဲ့ ရှာဖွေတာ ဖြစ်ပါတယ်။

* **`ausearch -k identity`**: Audit logs တွေထဲကမှ `identity` ဆိုတဲ့ key ပါတဲ့ ဖြစ်စဉ်တွေကိုပဲ ရှာပေးပါလို့ ခိုင်းတာပါ။ (User ဆောက်တာ၊ ဖျက်တာတွေက ဒီ key နဲ့ မှတ်တမ်းဝင်ပါတယ်)။
* **`| tail -n 10`**: Log တွေက အများကြီးမို့လို့ နောက်ဆုံးထွက်လာတဲ့ စာကြောင်း ၁၀ ကြောင်းကိုပဲ ကြည့်မယ်လို့ ဆိုလိုတာပါ။
---

## Testing and Verification (လက်တွေ့စမ်းသပ်ခြင်း)

ကျွန်ုပ်တို့ ထည့်သွင်းထားသော CIS Audit Rules များ တကယ်အလုပ်လုပ်ကြောင်း သိရှိရန် User အသစ်တစ်ခု ဖန်တီးပြီး စမ်းသပ်နိုင်ပါသည်။

### **Step 1: Create a Test User**

User အသစ်တစ်ခုကို ဆောက်ကြည့်ပါ။ ဤလုပ်ဆောင်ချက်သည် `/etc/passwd` ကို ပြောင်းလဲစေသဖြင့် Audit Log ထွက်လာရပါမည်။

```bash
sudo useradd testuser
```

### **Step 2: Check Audit Logs**

ကျွန်ုပ်တို့ သတ်မှတ်ထားသော `identity` ဆိုသည့် Key ကို အသုံးပြု၍ Log ထွက်၊ မထွက် စစ်ဆေးပါ။

```bash
sudo ausearch -k identity | tail -n 10
```
**Expected Result:** Log ထဲတွင် `testuser` ကို ဖန်တီးခဲ့သည့် မှတ်တမ်းနှင့် အသုံးပြုခဲ့သည့် command (exe="/usr/sbin/useradd") ကို တွေ့ရမည်ဖြစ်ပါသည်။ ဤသည်မှာ ကျွန်ုပ်တို့၏ CIS Hardening အောင်မြင်ကြောင်း သက်သေပြချက် ဖြစ်သည်။

### **နည်းပညာပိုင်းဆိုင်ရာ အကြံပြုချက်:**

ရိုက်ထားတဲ့ `tail -l` ဆိုတာလေးက `tail -n 10` (Line နံပါတ်) သို့မဟုတ် `tail -10` ဖြစ်ရပါမယ်။ `l` (အယ်လ်အသေး) ဆိုရင် command က error ပြနိုင်လို့ပါခင်ဗျ။

## Summary

`sudo aureport --summary` ဆိုတာကတော့ audit logs တွေ အများကြီးထဲကနေပြီး အရေးကြီးတဲ့ အချက်အလက်တွေကို **ဇယားနဲ့ အနှစ်ချုပ် (Summary Report)** ပုံစံ ထုတ်ကြည့်တာ ဖြစ်ပါတယ်။

ဒါဟာ SOC Analyst တစ်ယောက်အတွက် တစ်နေ့တာအတွင်း ဘာတွေထူးခြားလဲဆိုတာကို မျက်စိတစ်ဆုံးနဲ့ မြန်မြန်ဆန်ဆန် စစ်ဆေးနိုင်တဲ့ အသုံးဝင်ဆုံး Command တစ်ခု ဖြစ်ပါတယ်။

### **`sudo aureport --summary` ၏ လုပ်ဆောင်ချက်**

ဒီ command ကို ရိုက်လိုက်ရင် Audit system က အောက်ပါ အချက်အလက်တွေကို စာရင်းပြုစုပေးမှာ ဖြစ်ပါတယ်-

* **Users:** ဘယ် user တွေက system ထဲမှာ အလုပ်လုပ်ခဲ့သလဲ။
* **Terminals:** ဘယ်နှစ်ကြိမ် login ဝင်ခဲ့သလဲ။
* **Commands:** ဘယ်လို command တွေကို အသုံးများခဲ့သလဲ။
* **Files:** ဘယ်ဖိုင်တွေကို ဝင်ရောက်ကြည့်ရှု/ပြင်ဆင်ခဲ့သလဲ။
* **Authentication:** Login အောင်မြင်မှုနဲ့ ကျရှုံးမှု ဘယ်နှစ်ကြိမ်စီ ရှိသလဲ။
* **Syscalls:** Kernel level မှာ ဘယ်လို လုပ်ဆောင်ချက်တွေ အများဆုံး ဖြစ်ခဲ့သလဲ။

---

### Generating Executive Summary Report**

Audit Logs များကို တစ်ခုချင်းစီ လိုက်မဖတ်ဘဲ တစ်နေ့တာအတွင်း ဖြစ်ပျက်ခဲ့သည့် အခြေအနေများကို အကျဉ်းချုပ် (Executive Summary) အနေဖြင့် ကြည့်ရှုလိုပါက အောက်ပါ command ကို အသုံးပြုနိုင်ပါသည်။

```bash
sudo aureport --summary
```

**အသုံးဝင်ပုံ:**
* Security Monitoring ပြုလုပ်ရာတွင် ပုံမှန်မဟုတ်သော User activity များ သို့မဟုတ် Command ရိုက်ချက်များကို တစ်ချက်တည်းဖြင့် မြင်နိုင်ခြင်း။
* Login အကြိမ်ကြိမ် မှားယွင်းမှု (Failed Authentications) ရှိ၊ မရှိကို အမြန်စစ်ဆေးနိုင်ခြင်း။
---

### (Additional Reports):**

`--summary` အပြင် အခြား အသုံးဝင်တဲ့ Report အမျိုးအစားလေးတွေလည်း ရှိပါသေးတယ်-

* `sudo aureport -u`: ဘယ် User တွေ ဘာလုပ်သလဲဆိုတာပဲ သီးသန့်ကြည့်ချင်ရင်။
* `sudo aureport -f`: ဖိုင်တွေအပေါ် လုပ်ဆောင်ချက်တွေကိုပဲ သီးသန့်ကြည့်ချင်ရင်။
* `sudo aureport -au`: Login အောင်မြင်/ကျရှုံးတာတွေကိုပဲ (Authentication report) ကြည့်ချင်ရင်။
